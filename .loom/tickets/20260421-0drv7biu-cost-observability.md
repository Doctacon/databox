# Goal

Emit a daily MotherDuck usage summary (rows scanned, compute time, storage size) as Dagster asset metadata on a new `analytics.mart_cost_summary` asset. Auto-render to `docs/cost.md` in the deployed docs site. Local-backend users see equivalent local-resource-use numbers (DuckDB file size, query counts).

# Why

A "production-quality scaffold" that doesn't answer "how much is this costing me?" is incomplete. MotherDuck bills on compute-seconds and storage; a forker running the scaffold cloudward should see their spend trend without logging into the MotherDuck console.

This is also a portfolio-relevant signal â€” "cost awareness" is named in the staff-portfolio-readiness initiative's success criteria. It wasn't addressed there; it belongs here in the operability phase.

# In Scope

- New SQLMesh model: `analytics.mart_cost_summary`
  - On `motherduck` backend: queries MotherDuck's `system.catalog.*` (or equivalent usage views) to get daily query count, compute-seconds consumed, storage bytes
  - On `local` backend: queries SQLite Dagster event log + filesystem `du` via a Python-model kind if MotherDuck-style metadata is unavailable; at minimum emits daily materialisation count and warehouse-file size
  - Grain: one row per day
  - Columns: `day_date`, `backend`, `query_count`, `compute_seconds`, `storage_bytes`, `motherduck_cost_usd` (nullable on local)
- Dagster asset wrapping the model; writes freshest row's values to asset metadata (visible in Dagster UI asset page)
- `scripts/render_cost_page.py` â€” queries the mart, emits `docs/cost.md` with a time-series chart (matplotlib â†’ static PNG under `docs/images/cost/`) and a 30-day table
- Scheduled daily via Dagster
- Link from README + docs nav
- `docs/cost.md` explains the cost model per backend (forker understands what the numbers mean)

# Out of Scope

- Fine-grained per-query cost attribution (MotherDuck doesn't expose this cleanly)
- Budget alerts (forker plugs their own; document the extension point in `docs/cost.md`)
- Per-source cost attribution beyond what `query_id â†’ model` correlation naturally gives
- Cost forecasting

# Acceptance Criteria

- `analytics.mart_cost_summary` materialises successfully on both backends
- Running the asset on `motherduck` produces non-null `compute_seconds`, `storage_bytes`, `motherduck_cost_usd`
- `docs/cost.md` is auto-generated by `scripts/render_cost_page.py` with a 30-day chart and table
- Dagster asset metadata visibly shows current day's usage in the asset page sidebar
- Daily schedule runs the asset and re-renders the docs page
- MkDocs strict build clean with the new page

# Approach Notes

- MotherDuck exposes usage views; check current API surface. If unavailable programmatically, fall back to a Python model that shells out to the `motherduck` CLI
- For local backend, `os.stat(database_path).st_size` gives storage; query count approximates from Dagster run_log
- Keep the cost model thin: this is "visibility", not "optimisation". Optimisations are a future concern
- Currency conversion: hardcode MotherDuck's public per-compute-second rate in a constant, document it, note that forkers should update when MD pricing changes

# Evidence Expectations

- Asset materialisation log on both backends
- Rendered `docs/cost.md` with actual numbers in the deployed docs site
- Dagster asset-page screenshot showing metadata
- A PR-description paragraph explaining cost-model assumptions and how a forker adjusts them
